# 售后工作流

> **核心原则**：售后修改只做增量，不全盘重做。每次售后在项目目录下 `售后/` 文件夹中建日期+内容子目录。

## 售后文件夹结构

```
项目目录/
├── 原始数据/
├── 交付成果/          ← 首次交付的最终版
├── 售后/
│   ├── 2026-03-30_修改表格格式/
│   │   ├── 修改说明.md
│   │   └── (修改后的文件)
│   ├── 2026-04-02_补充回归分析/
│   │   ├── 修改说明.md
│   │   └── ...
│   └── ...
└── *.py
```

## 售后触发工作流

```
Step 1 【接到售后需求】
       ├─ 在 `售后/` 下创建子文件夹：`YYYY-MM-DD_售后内容简述/`
       ├─ 记录客户需求到 `修改说明.md`
       └─ 记录到 Obsidian

Step 2 【判断修改类型】（复用 I_modification.md 的 RED/YELLOW/GREEN/BLUE 分类）
       ├─ GREEN → 只改对应位置，直接输出到售后文件夹
       ├─ YELLOW → 修改脚本对应步骤，重跑受影响部分
       ├─ RED → 全盘重跑
       └─ BLUE → 新建脚本/步骤，输出到售后文件夹

Step 3 【执行修改】
       ├─ 修改后的文件放到 `售后/YYYY-MM-DD_xxx/` 下
       ├─ **不覆盖** `交付成果/` 中的原始交付文件
       └─ 如需更新交付成果，先备份再替换（带日期后缀）

Step 4 【验证 + 记录】
       ├─ 验证修改结果正确
       ├─ 更新 `修改说明.md` 记录执行结果
       └─ 追记 progress.md + Obsidian
```

## 修改说明.md 模板

```markdown
# 售后修改记录

## 客户需求
- [具体要求描述]

## 修改类型
- [ ] GREEN - 格式/文字修改
- [ ] YELLOW - 分析参数修改
- [ ] RED - 数据/逻辑修改
- [ ] BLUE - 新增分析

## 修改内容
- [具体改了什么]

## 输出文件
- [生成/修改的文件列表]

## 完成状态
- [ ] 已完成
- [ ] 已验证
```

## 铁律

1. **售后不全盘重做**：除非修改类型为 RED
2. **不覆盖原交付**：修改后的文件放售后文件夹
3. **每次售后必建文件夹**：`售后/YYYY-MM-DD_内容/`
4. **必须记录**：每次售后都要写 `修改说明.md` + Obsidian
5. **项目日志**：每个项目文件夹必须有 `项目日志.md`，简要记录开始日期、交付日期、售后日期及主题
